## What was done

### Part 1: CI Workflow

Created `.github/workflows/ci.yml` with two parallel jobs:

**Frontend job** (`frontend`):
- Checks out code, sets up Node 20 with npm cache
- `npm ci` -- install dependencies
- `npm run check` -- Svelte/TypeScript type checking
- `npm run lint` -- ESLint
- `npm run format:check` -- Prettier
- `npm run build` -- Vite production build

**Rust job** (`rust`):
- Checks out code, installs Tauri system dependencies (libwebkit2gtk-4.1-dev, libgtk-3-dev, libayatana-appindicator3-dev, librsvg2-dev, patchelf)
- Sets up Rust stable with rustfmt and clippy components
- Caches cargo registry, git sources, and `src-tauri/target/`
- Builds the frontend first (required by Tauri build process)
- `cargo fmt --check` -- formatting
- `cargo clippy -- -D warnings` -- lint with warnings as errors
- `cargo test` -- run all workspace tests

Triggers: push to main/master, pull requests targeting main/master.

## Design decisions

1. **Two separate jobs rather than one**: Frontend and Rust checks run in parallel for faster CI feedback. They have different system dependency requirements, so separation is cleaner.

2. **Node 20 pinned in CI**: Even though the local dev environment has Node 25, CI uses Node 20 as the minimum supported version per project requirements. This catches compatibility issues early.

3. **Rust job builds frontend first**: Tauri's build process expects the frontend dist directory to exist. Without it, clippy and cargo test would fail on the Tauri crate.

4. **Capability minimalism**: Only added core:window:default and core:event:default. These are foundational permissions any Tauri app needs. File system, dialog, and shell permissions are deferred to Epic 1 when the backend filesystem service is implemented.
