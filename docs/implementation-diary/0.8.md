# Task 0.8: Define IPC Contracts and Workspace Data Model Docs

## Date
2026-03-10

## Objective
Review, complete, and ensure consistency of the IPC contracts and workspace data
model specification documents so they are implementation-ready for Epic 1 and
Epic 2 work.

## What Already Existed
Both `docs/specs/ipc-contracts.md` and `docs/specs/workspace-data-model.md`
existed with reasonable scaffolds:
- IPC contracts had all MVP command names listed with brief return descriptions
- Data model had FileEntry, FuzzyMatch, NoteNode, LinkEdge, GitHunk, and
  WorkspaceStatus structures with field names (some without types)
- Versioning, error codes, event channels, and change control sections present

## Gaps Identified

### IPC Contracts
1. Commands had prose descriptions but no formal type signatures or parameter tables
2. Missing response type schemas (fields listed inline without types)
3. Missing commands: `rename_file`, `delete_file` (present in 2.7 guidance)
4. No error scenarios listed per command
5. Event channels used non-standard naming (`index://updated` vs `vigil://` prefix)
6. No payload schemas for events (just prose descriptions)
7. No payload size limits documented
8. No performance budget cross-references
9. No frontend wrapper module mapping
10. Error code table incomplete (missing `FILE_ALREADY_EXISTS`, `BINARY_FILE`, `STALE_ETAG`)

### Workspace Data Model
1. Missing `Vault` concept (required by task spec)
2. Missing `Note`/`NoteMetadata` as a distinct model from generic FileEntry
3. Missing `Tag` as a first-class entity model
4. Missing `DirEntry` (frontend-facing subset of FileEntry)
5. Field types inconsistent: some models had full types, others just field names
6. Missing event payload models (IndexChange, IndexReady, FsRenamed)
7. Missing cross-reference to IPC contracts
8. Missing tag extraction and title extraction rules
9. No serialization notes (e.g., `absolute_path` being `#[serde(skip)]`)

## Changes Made

### `docs/specs/ipc-contracts.md` (complete rewrite)
- Added formal function signatures with typed parameters and named response types
- Added full parameter tables and response field tables for every MVP command
- Added `rename_file` and `delete_file` commands
- Added per-command error lists and implementation notes
- Added performance budget cross-references for latency-sensitive commands
- Expanded error code table with `FILE_ALREADY_EXISTS`, `BINARY_FILE`, `STALE_ETAG`
- Added payload size limits table
- Standardized event channel names to `vigil://` prefix convention
- Added `vigil://index-ready` event for initial scan completion
- Added `vigil://fs-renamed` event for rename operations
- Added full typed schemas for all event payloads with common metadata fields
- Added event channel naming convention section
- Added frontend IPC wrapper module mapping table
- Expanded Epic 4 command stubs with response type names
- Added contract_version field in common envelope description

### `docs/specs/workspace-data-model.md` (complete rewrite)
- Added `Vault` model bridging workspace to logical namespace
- Added `NoteMetadata` model with title/tag extraction rules
- Added `Tag` as a first-class entity with count and files
- Added `DirEntry` as frontend-facing subset of FileEntry
- Added full type annotations to every field in every model
- Added event payload models: `IndexChange`, `IndexReady`, `FsRenamed`
- Added serialization notes (serde skip for absolute_path)
- Added cross-reference section linking to IPC contracts
- Added detailed caching strategy with persisted cache table
- Added tag extraction rules (frontmatter + inline) and title extraction rules
- Added concurrency model table with thread/sync mapping
- Expanded ignore policy with precedence order

## Consistency Checks
- Every response type referenced in IPC contracts has a matching model definition
  in workspace data model
- Event payload types match between both documents
- Field names and types are identical across both documents
- Error codes referenced in IPC command sections exist in the error code table
- `WorkspaceStatus` fields match between IPC response and data model
