# Task 0.9: Author architecture and UI mock/component map docs

## Date: 2026-03-10

## Summary

Reviewed, completed, and hardened the two architecture documents to be coherent and
implementation-ready. Both documents previously existed as stubs with correct structure
but insufficient detail for direct implementation use.

## Changes

### `docs/architecture/01-repository-architecture.md`
- Expanded Top-Level Structure tree to include all file-level entries (component files,
  Rust source files, config files) matching both the actual on-disk layout and the
  canonical plan from `vigil-plan/ARCHITECTURE.md`.
- Converted Ownership Rules from a flat bullet list to two detailed tables (Backend and
  Frontend) with path, owner, and responsibility columns.
- Added Cross-Cutting Rules section codifying: no business logic in commands, no direct
  Tauri calls from components, Rust-to-TypeScript type flow, feature isolation, and
  plugin scope constraints.
- Expanded UI Composition Target with an ASCII zone diagram, a zone definition table
  mapping each zone to layout component and feature module, and layout behavior notes.
- Added Data Flow diagram showing the full path from UI components through stores, IPC
  layer, Tauri bridge, command handlers, and core services to filesystem.
- Added Concurrency Model section referencing file watcher threading, lock-protected
  reads, per-file mutation locks, and serialized event delivery.
- Added Key Dependencies tables for Rust crates and frontend packages.
- Added Design Tokens section with CSS `@theme` reference.
- Added Scope Exclusions section explicitly stating no Vim mode and no plugin marketplace.
- Added Cross-References table linking to all related spec documents.

### `docs/architecture/02-ui-mock-component-map.md`
- Expanded Mock Zones from a brief list to detailed per-zone subsections with visual
  description, keyboard access, data sources, and interaction patterns.
- Restructured Component Mapping into categorized tables: Layout Components, Chrome
  Components, and per-feature-domain tables (Explorer, Editor, Preview, Graph, Omnibar,
  Status, Links, Plugins).
- Expanded Backend Capability Mapping into three tables: MVP Commands, Event Channels,
  and Epic 4 Commands/Events. Each entry now includes zones served, frontend consumer,
  and description.
- Added Store-to-Zone Mapping table showing which stores feed which zones.
- Added IPC Flow per Zone section with sequence diagrams for Explorer, Editor, Omnibar,
  and Status Bar interaction patterns.
- Expanded Theme Tokens into a full table with token, value, and usage columns.
- Added Scope Exclusions section (no Vim, no marketplace).
- Added Cross-References table.

### `docs/README.md`
- Added Implementation Diary section referencing the `implementation-diary/` directory.

## Decisions

- Reconciled the two competing folder structures (the existing `01-repository-architecture.md`
  stub and `vigil-plan/ARCHITECTURE.md`) by keeping the features-based organization from the
  stub (which matches the actual directory layout) while incorporating file-level detail from
  the vigil-plan reference.
- Component paths in the UI map use the `features/` organization consistently, not the flat
  `components/` layout from `vigil-plan/ARCHITECTURE.md`, since the actual repo follows the
  features pattern.
- Epic 4 content (graph improvements, plugin runtime, content search) is included but clearly
  labeled as Epic 4 to avoid scope confusion during MVP implementation.
