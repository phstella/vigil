## What was done

Implemented the workspace status service that aggregates git branch/sync state, index metrics (notes, tags, files counts), and app version into a single `WorkspaceStatus` response. Added the `workspace_status` Tauri command, a status event emitter, and integration tests.

### Key design decisions

1. **Separate `core/git/status.rs` from `core/git/service.rs`**: The existing `GitService` handles diff hunks and file-level status. Branch/sync queries are conceptually different (repo-level, not file-level), so they live in a dedicated module as free functions rather than extending `GitService`.

2. **Free functions vs struct**: `current_branch()` and `sync_state()` are stateless functions that take a `&Path` and use `Repository::discover()`. No need for a struct since they don't hold state -- each call opens the repo fresh, which is fast for these lightweight queries.

3. **Sync state logic**: For MVP, sync state checks both upstream ahead/behind counts AND uncommitted changes. If there's no upstream configured and the tree is clean, it reports `Synced`. Dirty tree with no upstream reports `Unknown`. When upstream exists, standard ahead/behind/diverged logic applies.

4. **Metrics collection**: `collect_metrics()` takes `Option<&FileIndex>` and `Option<&TagIndex>` to gracefully handle the case where no workspace is open or indices aren't built yet, returning zero counts.

5. **Version from Cargo.toml**: Uses `env!("CARGO_PKG_VERSION")` compile-time macro rather than reading Cargo.toml at runtime. This is idiomatic Rust and guaranteed to match the actual build version.

6. **Event emitter ready but not auto-triggered**: `emit_status_updated()` is implemented and available for use by the watcher or other components that detect status changes. The command itself does not emit events -- it's a pull-based query. Push-based status updates can be wired in when the file watcher or git polling is enhanced.
