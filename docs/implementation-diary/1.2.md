### Derive traits

All structs derive `Serialize, Deserialize, Clone, Debug` at minimum. Enums additionally derive `Copy, PartialEq, Eq` where appropriate.

### Error handling design

- `VigilError` uses `thiserror::Error` derive for `Display` + `Error` impls
- 12 error variants match the IPC contract error code table 1:1
- `ErrorCode` serializes as `SCREAMING_SNAKE_CASE` to match the frontend contract
- `ErrorEnvelope` provides the `{ code, message, details }` JSON shape
- `From<VigilError> for tauri::ipc::InvokeError` allows `Result<T, VigilError>` as direct command return type
- Convenience `From` impls for `std::io::Error` and `git2::Error` map common library errors to appropriate variants

### serde conventions

- Enums like `EntryKind`, `HunkChangeType`, `LinkType`, `SyncState` use `#[serde(rename_all = "lowercase")]`
- `ErrorCode` uses `#[serde(rename_all = "SCREAMING_SNAKE_CASE")]`
- `FileEntry.absolute_path` uses `#[serde(skip)]` per the data model spec
- `ErrorEnvelope.details` uses `#[serde(skip_serializing_if = "Option::is_none")]`

## Design decisions

1. **i64 for timestamps, u64 for counts/sizes**: Timestamps use `i64` to match `chrono::DateTime<Utc>.timestamp_millis()` return type. Counts and byte sizes use `u64` since they are never negative.

2. **String paths instead of PathBuf for IPC types**: Workspace-relative paths in IPC-facing structs are `String` (matching JSON serialization and the contract spec). Only `FileEntry.absolute_path` uses `PathBuf` since it is backend-only and skipped during serialization.

3. **Separate request/response types**: Even when a response is simple (e.g., `DeleteFileResponse` with just a `path` field), it gets its own named struct rather than returning a bare `String`. This makes the IPC contract self-documenting and allows adding fields without breaking changes.

4. **ContentMatch and NoteGraphResponse included now**: These are Epic 4 types but are defined in the IPC contracts and data model specs. Including them now means later epics can use these types without modifying the models module.

5. **GitFileStatus and GitStatusEntry added beyond spec minimum**: The workspace data model doesn't explicitly list per-file git status, but `GitFileStatus` is the natural companion to `GitHunk` for explorer tree decorations. Added proactively since the git service (task 1.6) will need it.
