**Date**: 2026-03-11

## What was done

Implemented the fuzzy file finder service for the omnibar filename mode (`Ctrl+P`). The service uses `nucleo-matcher` for high-performance fuzzy scoring against workspace-relative file paths from the `FileIndex`. A Tauri command exposes the service over IPC matching the `fuzzy_find` contract.

### Key design decisions

1. **nucleo-matcher configuration**: Uses `Config::DEFAULT.match_paths()` which sets `/` as a delimiter character and adjusts boundary bonuses for path matching. This gives appropriate scoring for path separators in workspace-relative paths.

2. **Pattern API**: Uses `nucleo_matcher::pattern::Pattern` with `CaseMatching::Smart` (case-insensitive when query is all lowercase, case-sensitive otherwise) and `Normalization::Smart` (unicode normalization). The `Pattern::indices()` method returns both the score and the matched character positions in a single pass.

3. **Empty query behavior**: Returns all files sorted by `modified_at_ms` descending (most recently modified first), capped to the requested limit. This provides a useful "recent files" list when the omnibar first opens.

4. **Directory exclusion**: Only files (not directories) are scored and returned, matching the omnibar's purpose of file navigation.

5. **Stateless design**: `FuzzyFinder` is created per-query with a borrowed `&FileIndex`. No persistent state is needed since `nucleo-matcher`'s `Matcher` is cheap to construct and the index snapshot is read atomically.

6. **Score type**: Match scores are stored as `f64` in `FuzzyMatch` (converted from nucleo's `u16` score). This matches the IPC contract definition and allows future scoring adjustments.
