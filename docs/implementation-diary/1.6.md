# Task 1.6: Implement Content Search Index/Service

**Date**: 2026-03-11
**Status**: DONE

## What was done

Implemented the in-memory content search service for phrase/snippet matching within file contents. The service reads files from disk using the `FileIndex` to enumerate workspace files, performs case-insensitive substring matching, and returns ranked `ContentMatch` results with file path, line number, column ranges, and preview snippets. A Tauri command exposes the service over IPC matching the `search_content` contract.

### Key design decisions

1. **Simple in-memory grep approach**: For MVP, the searcher reads each file from disk on every query rather than maintaining a persistent full-text index. This keeps the implementation simple and avoids heavy dependencies like tantivy. Files are read via `fs::read_to_string` and searched line-by-line.

2. **Binary file and size guards**: Files flagged as binary by the `FileIndex` are skipped entirely. Files larger than 2 MB are also skipped to avoid excessive memory consumption during search.

3. **Multiple matches per line**: When a query appears multiple times on the same line, each occurrence produces a separate `ContentMatch` with distinct `line_start_col`/`line_end_col` values. This gives the frontend precise highlight information.

4. **Relevance scoring**: A simple heuristic scoring function considers: match at start of trimmed line (+20), exact case match bonus (+5), line brevity bonus (shorter lines score higher since the match is more prominent), and query length bonus (longer queries are more specific). Results are sorted by score descending with path and line number as stable tiebreakers.

5. **Deterministic file ordering**: Files are sorted by path before searching, ensuring reproducible result ordering when scores are equal.

6. **Workspace root from FileIndex**: The `search_content` command gets the workspace root from `FileIndex::root()` rather than requiring an additional state lookup, since the index already knows its root path.

7. **Stateless design**: Like `FuzzyFinder`, `ContentSearcher` is created per-query with a borrowed `&FileIndex`. No persistent search state is maintained.
