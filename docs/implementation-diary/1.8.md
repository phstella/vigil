# Task 1.8: Link Graph and Backlinks Resolver

**Date**: 2026-03-11
**Status**: DONE
**Branch**: epic0-claude

## What was done

Implemented the link parser, in-memory bidirectional link graph, backlinks command, and integration tests. The system parses `[[wikilinks]]` and `[text](path.md)` markdown links from note content, builds forward and reverse link indices, and exposes a `get_backlinks` Tauri command matching the IPC contract.

### Key design decisions

1. **Parser architecture**: Regex-based extraction with a code-block stripping pre-pass. Fenced code blocks (triple-backtick) are replaced with empty lines to preserve byte offsets for context extraction. Two separate regex passes for wikilinks and markdown links.

2. **Link type preservation**: `LinkType::Wikilink` vs `LinkType::Markdown` is tracked through the entire pipeline from parsing to edge storage, satisfying the requirement to preserve link type distinction.

3. **Dangling link handling**: Links to non-existent notes produce backlink records and graph edges. The `get_graph()` method creates placeholder `NoteNode` entries for targets that don't exist in the index, so they appear in the visualization graph.

4. **Path resolution**: Wikilinks are resolved as note stems (`.md` appended if no extension). Markdown links are resolved relative to the source note's directory, with `..` collapsing and `./` stripping.

5. **Context snippets**: The line containing each link is extracted and truncated to 200 characters, matching the `BacklinkRecord.context_snippet` contract.

6. **Node IDs**: Deterministic 16-hex-char IDs derived from `DefaultHasher` hash of workspace-relative path. Stable across rebuilds.

7. **Graph lifecycle**: `LinkGraph` is built during `open_workspace` immediately after index scan. Stored in `AppState` alongside the index. Both are cleared on workspace switch.
