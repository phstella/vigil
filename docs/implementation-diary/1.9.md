# Task 1.9: Tag Extraction and Index

**Date**: 2026-03-11
**Status**: DONE
**Branch**: epic0-claude

## What was done

Implemented a dedicated in-memory tag index, two new Tauri commands (`get_all_tags`, `get_files_by_tag`), and code-block stripping for inline tag extraction. The tag index is built from the existing `FileIndex` note metadata and supports case-insensitive tag matching with sorted results.

### Key design decisions

1. **Separate TagIndex vs extending FileIndex**: Created a dedicated `TagIndex` struct rather than adding tag queries directly to `FileIndex`. This follows the same pattern as `LinkGraph` -- a derived data structure that rebuilds from the file index. Keeps `FileIndex` focused on file metadata extraction.

2. **Case-insensitive matching**: All tags are normalized to lowercase at storage time (in both `NoteMetadata.tags` during extraction and in `TagIndex` during rebuild). Query methods also normalize input to lowercase, ensuring `get_files_by_tag("Project")` matches `#project`.

3. **Code block stripping**: Added `strip_fenced_code_blocks()` to `service.rs` that handles both backtick and tilde fences with arbitrary fence lengths. Integrated into `extract_note_metadata()` before `extract_inline_tags()` is called, so tags like `#code-tag` inside fenced code blocks are never indexed.

4. **Tag model**: The `Tag` struct includes `files: Vec<String>` in addition to `count: u64`, matching the workspace data model spec. This allows the frontend to display which files use a tag without a separate query.

5. **Sort order**: `get_all_tags()` sorts by count descending, then name ascending for deterministic ordering. `get_file_tags()` and `get_files_by_tag()` return alphabetically sorted results.

6. **Lifecycle**: `TagIndex` is built during `open_workspace` immediately after index scan (and after link graph). Stored in `AppState` and cleared on workspace switch via `clear_all()`.
