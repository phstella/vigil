### Design decisions

- **svelte/store writable()**: Used `writable()` from `svelte/store` rather than Svelte 5 runes for stores. Runes (`$state`) only work inside `.svelte` files or `.svelte.ts` files, while `writable()` works in plain `.ts` files and provides cross-component reactivity that integrates cleanly with Svelte 5's `$` auto-subscription syntax.
- **No IPC or event listeners**: Stores define only state shape and synchronous mutation methods. IPC calls and event subscriptions will be wired in later tickets, keeping stores pure and testable.
- **Factory pattern**: Each store is created via a `createXxxStore()` function that returns the writable's `subscribe`/`set` plus domain-specific mutation methods. This keeps the public API clean while hiding the writable internals.
- **Set for expandedDirs**: Used `Set<string>` rather than an array for O(1) lookup/toggle of expanded directories in the file explorer.
- **Map for hunks**: Used `Map<string, GitHunk[]>` for per-file hunk storage, allowing efficient lookup by file path.
- **Partial settings update**: `settingsStore.updateSetting()` accepts `Partial<SettingsState>` so callers can update one or many fields in a single call.
- **Tab close behavior**: When closing the active file, the store selects an adjacent tab (or clears to null if no tabs remain), matching standard editor tab behavior.
