# Task 3.11 -- Performance Hardening Against Budget Gates

## 2026-03-12T00:00 -- Initial Plan

**Objective**: Cross-cutting performance hardening pass across features from tasks 3.1-3.10.

**Plan (6 steps)**:
1. Lazy-load Monaco: Ensure CodeEditor.svelte is dynamically imported in EditorRouter so the Monaco bundle is not included in the main chunk.
2. Debounce omnibar: Verify debounce in omnibar-store is correct. Add leading-edge fire for file mode to meet <=80ms first-result budget.
3. Virtualize the file tree: Implement windowed/virtualized rendering in FileTree.svelte so only visible rows are rendered (critical for 10k-file workspaces).
4. Add frontend performance instrumentation: Add perf timing helpers for omnibar open-to-first-paint, editor mount, and pane switch render.
5. Run all required checks (npm run check, lint, build, cargo test).
6. Fix any failures and finalize.

**Key budget targets**:
- Cold launch to interactive: <= 1200ms
- Ctrl+P first result: <= 80ms
- Keystroke-to-paint: <= 16ms p95
- File switch: <= 120ms
- Scroll FPS: >= 60 FPS
- Workspace open (10k files): <= 1500ms first usable tree

## 2026-03-12T00:10 -- Implementation Complete

### Step 1: Lazy-load Monaco via dynamic import in EditorRouter
- `EditorRouter.svelte` now uses `import('./CodeEditor.svelte')` (dynamic) instead of a static import.
- The `{#await}` block shows a loading state while Monaco loads, then renders the editor.
- Confirmed in `npm run build` output: `CodeEditor.js` is a separate server chunk.
- The static `CodeEditor` export was removed from EditorRouter; it stays in the barrel `index.ts`.

### Step 2: Omnibar leading-edge debounce
- `omnibar-store.ts`: Added `leadingEdgeFired` flag to fire the IPC call immediately on the first keystroke in file mode, meeting the <=80ms first-result budget.
- Subsequent rapid keystrokes are trailing-debounced at 80ms (file) / 150ms (content).
- Content mode remains trailing-only (results are heavier and 150ms budget is more lenient).

### Step 3: Virtualized file tree
- `FileTree.svelte`: Completely rewritten to flatten the tree and virtualize rendering.
- For trees with < 200 visible nodes, renders all nodes directly (no overhead).
- For trees >= 200 visible nodes, uses scroll-position-based windowing with a 28px row height, ResizeObserver for container measurement, and 10-row overscan.
- Removed self-recursive FileTree import; flattening handles depth natively.

### Step 4: Frontend performance instrumentation
- Created `src/lib/utils/perf.ts` with `perfTimer()`, `timeAsync()`, `timeSync()` utilities.
- Logs to console in dev mode and when `window.VIGIL_PERF` is set.
- Budget-aware: warns when a measurement exceeds its budget.
- Instrumented:
  - `CodeEditor.svelte` onMount: `perfTimer('editor-mount', 120)`
  - `Omnibar.svelte` mount: `perfTimer('omnibar-open', 80)`
  - `EditorRouter.svelte` file switch: `perfTimer('pane-switch-render', 120)`

### Step 5: Check results
- `npm run check`: PASS (0 errors, 2 pre-existing warnings in GraphView.svelte)
- `npm run lint`: PASS
- `npm run build`: PASS (Monaco chunk warning is pre-existing)
- `cargo test`: NOT RUN (cargo not installed in this environment)
- `rg` perf grep: PASS (instrumentation found in Rust backend + frontend)

### Decisions
- FileTree virtualization threshold set to 200 nodes (balances simplicity vs. perf).
- Row height fixed at 28px for predictable scroll behavior.
- Leading-edge debounce only for file mode (content mode uses trailing-only).
