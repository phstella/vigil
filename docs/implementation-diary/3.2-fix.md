# 3.2-fix: Fix critical note-switch data-loss path

## Initial Plan

The `$effect` in `NoteEditor.svelte` that handles `filePath` changes had a data-loss bug: when `noteStore.open(targetPath)` returned `false` (meaning the pre-switch save failed), the code fell through to `noteStore.load(targetPath, fallbackContent)`, which loaded the new file anyway and discarded the unsaved buffer.

The fix: when `open()` returns `false`, do NOT call `load()`. Instead, reset `loadedPath` to the previous value so the effect can retry on next navigation, and keep the current dirty buffer intact.

## What Changed and Why

**File:** `src/lib/features/editor/NoteEditor.svelte`

1. **Captured `previousPath`** before updating `loadedPath`, so we can restore it on failure.
2. **Removed the `noteStore.load()` fallback** from the `!success` branch. Instead, the handler logs a warning and resets `loadedPath = previousPath`, preserving the dirty buffer.
3. **Removed unused `fallbackContent` variable** (was only used by the now-removed `load()` call). Renamed `content` prop destructuring to `_content` with a `void` usage to satisfy the linter while keeping API compatibility.
4. **Removed stale comment** ("Fall back to direct load if IPC is unavailable") since the fallback behavior is no longer present.

## Handoff Notes

- The `content` prop is still accepted by the component for API compatibility but is no longer consumed. If callers are updated to stop passing it, the prop and its type can be removed.
- The Svelte warning `state_referenced_locally` on `_content` is benign -- we intentionally only reference it once via `void` to suppress the unused-var lint error.
- The `linksStore.setActivePath(filePath)` call still runs eagerly before the `open()` promise resolves. If the switch is blocked, backlinks may briefly show the wrong file's links. This is cosmetic and not a data-loss risk.
