# Implementation Diary: Task 3.2 -- Markdown Editing Lifecycle

## 2026-03-12T00:00 -- Initial Plan

**Goal:** Integrate markdown editing lifecycle (open/save/autosave/dirty state) connecting the NoteEditor component to the backend file service via IPC.

**Plan (5 steps):**

1. Enhance `note-store.ts` to track etag for optimistic concurrency, add autosave timer (debounced 2s), and wire save/load through IPC (`readFile`/`writeFile`).
2. Update `NoteEditor.svelte` to call `noteStore.open(path)` which loads via IPC, display a saving indicator, and trigger autosave on content changes.
3. Update `+page.svelte` to wire `Ctrl+S` to the IPC-backed `noteStore.save()` instead of the placeholder `markClean()`.
4. Run all required checks (`npm run check`, `npm run lint`, `npm run build`, `cargo test`).
5. Fix any failures and finalize diary.

**Key decisions:**
- Autosave debounce: 2 seconds after last keystroke (performance budget: file switch <= 120ms).
- Optimistic concurrency via etag from `ReadFileResponse` passed to `writeFile`.
- No Tiptap yet (that's task 3.3); keep textarea as editing surface.
- Save clears dirty state only on successful write_file response.

## 2026-03-12T00:03 -- Handoff Notes

**What was done:**
- Note-store now has a complete open/save/autosave lifecycle wired to backend IPC.
- Dirty state is tracked and only cleared after successful `write_file` response.
- Autosave fires 2 seconds after the last keystroke via debounced timer.
- Etag from `ReadFileResponse` is passed to `write_file` for optimistic concurrency (STALE_ETAG detection).
- When switching files while dirty, the current file is auto-saved first.
- NoteEditor shows a "saving" indicator during save and error text on failure.

**Risks / Follow-ups:**
- `cargo test` was not run because cargo is not installed in this sandbox. Must be verified in CI or locally.
- Autosave timer is per-store singleton; if multiple NoteEditor instances existed (not the case today), they would share state. This matches the single-note-pane architecture.
- STALE_ETAG conflict resolution (reload vs force-save) is not yet implemented in the UI; the error is displayed but user action is not prompted. This is a future task (ConflictBanner.svelte).
- The `content` prop on NoteEditor is kept for backward compatibility but is only used as a fallback if IPC open fails.
