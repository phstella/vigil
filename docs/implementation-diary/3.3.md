# Implementation Diary: Task 3.3 -- Live Markdown Render/WYSIWYG Toggle

## 2026-03-12T00:00 -- Initial Plan

**Ticket:** Implement live markdown render/WYSIWYG toggle while preserving raw text.

**Acceptance criteria (from canonical backlog + keyboard map + UI component map):**
- `Ctrl+.` toggles between WYSIWYG rendered view and raw markdown edit view.
- Raw text is always preserved (the source of truth is the raw markdown string).
- Toggle latency <= 100ms (performance budget).
- InlinePreview.svelte owns the rendered surface.
- NoteEditor.svelte integrates the toggle.

**Plan (5 steps):**
1. Create `src/lib/utils/markdown.ts` -- lightweight markdown-to-HTML renderer (headings, bold, italic, code, links, lists, blockquotes, horizontal rules, wikilinks).
2. Create `src/lib/features/preview/InlinePreview.svelte` -- renders HTML from markdown, styled with Vigil design tokens.
3. Extend `note-store.ts` with `viewMode` state ('edit' | 'preview') and toggle method.
4. Update `NoteEditor.svelte` to show raw textarea or InlinePreview based on viewMode, with a toggle button in the header.
5. Register `Ctrl+.` shortcut in `+page.svelte` for toggling, update barrel exports.

**Design decisions:**
- Used a dependency-free markdown renderer to avoid adding npm packages and keep toggle latency well within the 100ms budget. The renderer is synchronous and fast for typical note sizes.
- View mode persists across file loads (toggling to preview stays in preview when switching files). This matches user expectation.
- Raw text is never modified by the preview -- InlinePreview is read-only and derives HTML from the store's content.
- The `@html` directive is used with HTML-escaped source content (escapeHtml is applied before any inline rendering), mitigating injection risk.

**Handoff notes:**
- When Tiptap is integrated (future), InlinePreview can be replaced or layered with Tiptap's own rendering. The toggle mechanism and store state are ready for that.
- The markdown renderer handles the most common syntax but does not cover tables, footnotes, or task lists. These can be added incrementally.
- `cargo test` should be run when a Rust toolchain is available; no Rust files were modified.
