# Task 2.6: Svelte Stores for Application State

## Goal

Create reactive stores using Svelte 5 runes to manage editor, file tree, git,
and settings state across all components.

## Files to Create


## Design Decisions

- **Svelte 5 runes** (`$state`, `$derived`, `$effect`) instead of legacy
  `writable`/`readable` stores. Runes provide fine-grained reactivity and
  smaller compiled output.
- **Single state objects** exported per domain. Components import
  `editorState.activeFile` rather than individual stores.
- **Tags store**: populated from the Rust tag index (Task 1.8), updated
  incrementally on file-watcher events.
- **Settings persist** to `localStorage`. Other stores are session-only
  (reconstructed from Tauri commands on workspace open).
- **`fileDiffs` is a cache**: keyed by file path, populated lazily when a
  file is opened, invalidated on save or `fs-modified` events.
- **Activity bar state**: `activePanel` and `sidebarVisible` live in settings
  so they persist across sessions.

## Acceptance Criteria

- [ ] Changing `editorState.activeFile` triggers any component reading it to re-render
- [ ] `filesState.fileTree` is populated after calling `openWorkspace`
- [ ] `gitState.branch` reactively updates the status bar
- [ ] `settingsState.fontSize` persists to `localStorage` and loads on app restart
- [ ] `settingsState.activePanel` switches the sidebar panel content
- [ ] `tagsState.allTags` is populated after workspace open
- [ ] `tagsState.activeFileTags` updates when switching files
- [ ] All stores are importable from any `.svelte` or `.ts` file via `$lib/stores/*`
- [ ] TypeScript compiles in strict mode with all store types correct
- [ ] No circular dependencies between store files
