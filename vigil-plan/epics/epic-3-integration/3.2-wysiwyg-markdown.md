# Task 3.2: Inline WYSIWYG Markdown Editor

## Goal

Build an inline-rendered Markdown editing experience where formatting (headings,
bold, blockquotes, links) renders directly as the user types — while keeping the
raw Markdown accessible. This is the primary editing surface for `.md` files.

## Design Philosophy

This is **not** a split-pane preview. It's a single editor where:
- `# Heading` renders as a large styled heading in real time.
- `**bold**` renders as **bold** inline (markers fade/shrink but remain accessible on cursor focus).
- `> blockquote` renders with a teal left border bar.
- `[[wikilink]]` renders as a styled teal link.
- Code blocks render with syntax highlighting and a subtle background.
- Placing the cursor on any formatted element reveals its raw Markdown syntax.

This follows the Obsidian/Typora model: WYSIWYG until you focus the line,
then raw Markdown is visible for editing.

## Approach Options

### Option A: Monaco with Custom Decorations (Simpler MVP)

Use Monaco as the base, but heavily customize it with:
- **ViewZones** to inject rendered HTML above/below Markdown elements.
- **Decorations** to style headings (larger font), bold, italic, etc.
- **Content widgets** for inline-rendered elements.
- A CSS layer that hides Markdown syntax markers (`#`, `**`, `>`) when the
  cursor is not on that line, and reveals them on focus.

Pros: Leverages Monaco's editing engine, undo/redo, selection, etc.
Cons: Fighting Monaco's design — may feel hacky, limited control.

### Option B: Custom Prosemirror/Tiptap Editor (Richer, More Work)

Use a rich-text editing framework:
- **Tiptap** (built on ProseMirror) with a Markdown serializer.
- The internal model is rich-text, but saves as Markdown.
- Full control over rendering: headings, blockquotes, lists, code blocks.
- Markdown source mode toggle for power users.

Pros: True WYSIWYG, clean architecture, extensible.
Cons: Larger dependency, more complex integration.

### Recommended: Option B (Tiptap)

Tiptap provides the best UX for the Obsidian-like inline rendering described
in the design doc. Monaco is better suited for code files (Task 3.1).

## Files to Create

### `src/lib/components/editor/WysiwygEditor.svelte`

**Initialization:**
```ts
import { Editor } from '@tiptap/core';
import StarterKit from '@tiptap/starter-kit';
import Link from '@tiptap/extension-link';
import { WikilinkExtension } from './extensions/wikilink';

let editor: Editor;

onMount(() => {
  editor = new Editor({
    element: editorContainer,
    extensions: [
      StarterKit.configure({
        heading: { levels: [1, 2, 3, 4] },
        codeBlock: { /* syntax highlighting config */ },
      }),
      Link,
      WikilinkExtension,
      // Custom extensions for Vigil
    ],
    content: editorState.content,
    editorProps: {
      attributes: {
        class: 'vigil-prose',
      },
    },
    onUpdate: ({ editor }) => {
      editorState.content = editor.storage.markdown.getMarkdown();
      editorState.isDirty = true;
    },
    onSelectionUpdate: ({ editor }) => {
      const { from } = editor.state.selection;
      const pos = editor.view.coordsAtPos(from);
      // Update cursor position in store
    },
  });
});
```

**Markdown serialization:**
The editor works with a rich-text internal model but serializes to/from
Markdown for storage. Use `@tiptap/pm` with a Markdown parser/serializer:
- On file open: parse Markdown → ProseMirror document.
- On save: serialize ProseMirror document → Markdown string.
- The user always sees rendered content, but the file on disk is pure `.md`.

### `src/lib/components/editor/extensions/wikilink.ts`

Custom Tiptap extension for `[[wikilinks]]`:
- **Input rule**: typing `[[` triggers an inline autocomplete popup.
- **Node type**: renders as `<a class="wikilink">` in the editor.
- **Display**: shows the link text in teal with dotted underline.
- **Serialization**: outputs `[[target]]` or `[[target|display]]` in Markdown.

  for notes, and a crisp monospace font for code blocks."

## Integration with Editor Routing

In the parent `Editor.svelte` or `SplitPane.svelte`, choose which editor
to render based on file type:

```svelte
{#if activeFileExtension === '.md'}
  <WysiwygEditor />
{:else}
  <MonacoEditor />
{/if}
```

This gives `.md` files the WYSIWYG experience and all other files the
Monaco code editor (Task 3.1).

## Acceptance Criteria

- [ ] Opening a `.md` file renders formatted Markdown (headings, bold, lists, blockquotes)
- [ ] Headings render large and styled — `# Title` shows as a large bold heading with teal underline
- [ ] Blockquotes show a teal left border bar (matching the mock)
